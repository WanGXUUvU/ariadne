# TASK-038-2 - Run 可观测性（状态机 + Tool call 记录 + 超时）

## 目标
建立 Run 级别的可观测性：每次 run 有明确状态，每个工具调用有独立记录，工具执行有超时保护。

## 产品层
Runtime / Storage / Infrastructure

## 背景
TASK-038-1 完成 async 重构后，runtime 具备了中断能力。
但数据库里缺少结构化的 run 状态和工具执行记录，无法回溯"这次 run 调了什么工具、哪个成功、哪个被取消"。

## 用户动作
用户查看历史对话，展开某条 run 的详情。

## 用户会看到
- 每条 run 有状态标记：completed / cancelled / failed / timeout
- 可以看到这条 run 调用了哪些工具，每个工具的状态和结果
- 工具超时时，run 状态变为 timeout，不会无限等待

## 范围内
- DB 新增 `run_status` 字段（`SessionRunRecord`）：`running / completed / cancelled / failed / timeout`
- DB 新增 `ToolCallRecord` 表：`run_id, tool_name, status, input_json, result_json, started_at, finished_at`
- `agent_runtime.py`：工具执行前写 running 记录，执行后更新状态
- `run_service.py`：stream 开始时写 `status=running`，结束时更新 `status=completed`，CancelledError 时更新 `status=cancelled`
- 工具超时：`asyncio.wait_for` 包裹工具执行，超时触发 `status=timeout`
- alembic migration：新增字段和表
- API：新增 `GET /sessions/{id}/runs/{run_id}` 查询 run 详情（含 tool_calls）

## 范围外
- 前端展示 run 详情（单独前端卡）
- Run 级别的重试机制
- Tool call 级别的重试机制

## 实现步骤
1. 设计 `ToolCallRecord` 表结构和 `run_status` 字段
2. alembic migration
3. `agent_runtime.py`：工具执行前后写 ToolCallRecord
4. `run_service.py`：run 状态机更新
5. 工具超时：`asyncio.wait_for` + timeout 配置
6. API：`GET /sessions/{id}/runs/{run_id}`
7. 单元测试
8. 全量测试通过

## 完成标准
- DB 里每条 run 有 status 字段
- DB 里每次工具调用有独立记录
- 工具执行超时后 run 状态变 timeout
- GET run 详情接口返回 tool_calls 列表

## 验证
- `python3 -m unittest discover -s backend/tests -p 'test_*.py' -v`
- 手动跑一次含工具的对话，查询 DB 验证 tool_call_records
- 手动触发超时，验证 status=timeout

## Review 检查点
- migration 是否可回滚
- ToolCallRecord 是否覆盖工具失败场景
- 超时时间是否可配置（不写死）
- GET 详情接口是否做了权限校验（session_id 归属）
